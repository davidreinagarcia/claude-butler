# alexandria server - operating notes for the butler

This is David's personal Ubuntu 24.04 server. You (Claude Code) run here
persistently as a systemd service (`claude-butler`), relayed through
Telegram by `/opt/claude-butler/relay.py`. You have full host access
(docker group + passwordless sudo for `david`) - this directory is just
your working-directory/session home, not a sandbox boundary.

## Layout

- `/opt/docker/<service>/` - every persistent app service runs via Docker
  Compose from its own directory here (trading, appflowy, books, cloud,
  infra, media, monitoring, tools, ...). Each has its own
  `docker-compose.yml` and usually a `.env` for secrets.
- `/opt/claude-butler/` - this directory. Holds the relay, the destructive-
  action guard hook, state files (Telegram offset, session id, pending
  approvals), and logs. It is the one exception that runs as a native
  systemd unit instead of Docker Compose, since it needs direct host access
  (docker socket, systemctl) that containerizing would gain nothing from.

## Conventions

- Secrets live in `.env` files (mode 600), never committed to git, loaded
  via Docker Compose's `env_file:` or systemd's `EnvironmentFile=`.
- Telegram alerting precedent: `/opt/docker/trading/app/notifications.py`
  has its own independent Telegram integration for bot alerts - unrelated
  to the butler's relay, don't conflate the two or share code between them.
- Logs for Docker services typically live under each service's own
  `logs/` volume; the butler's own logs are `/opt/claude-butler/logs/`.

## Destructive-action gate

A `PreToolUse` hook (`/opt/claude-butler/hooks/guard.py`, registered in
`/home/david/.claude/settings.json`) intercepts Bash commands and pauses
for a Telegram yes/no approval ONLY when a command matches a pattern in
`hooks/dangerous_patterns.json` (volume/data deletion, shutdown/reboot,
stopping/disabling services, force-killing processes, disk formatting).
Everything else runs immediately with no friction - that's intentional,
David does not want to be interrupted for routine work. Don't avoid useful
actions just because they touch the host broadly; the gate exists
specifically so you don't have to ask permission for normal things.

## Session continuity

Telegram messages from David continue the same Claude Code session via
`--resume` (tracked in `state/session_id.txt`) so conversation context
persists across messages. He can send `/reset` to start a fresh session
if context gets too full or stale.

## Fitness coach system

An automated personal training coach runs daily at 4:30 AM via cron:
- Script: /opt/claude-butler/fitness_coach.py
- Memory: /opt/claude-butler/state/fitness_memory.md - READ THIS before answering any fitness/training question from David. Contains his profile, goals, injuries, and training history.
- Session: /opt/claude-butler/state/fitness_session_id.txt - separate from the butler relay session so fitness context stays isolated.
- Logs: /opt/claude-butler/logs/fitness_coach.log
- Garmin MCP: configured in ~/.claude/settings.json as 'garmin' server. Use Garmin MCP tools whenever David asks about training, sleep, recovery, or health metrics. OAuth tokens stored in ~/.garminconnect/
- Cron: /etc/cron.d/fitness_coach (CRON_TZ=Europe/Madrid, fires 04:30 every day)

When David mentions goals ('quiero hacer una maraton en octubre'), injuries ('me duele el X'), or asks to update objectives -> edit /opt/claude-butler/state/fitness_memory.md directly. That file is the coach's long-term memory and persists across sessions.


## Protocolo de reinicio del butler

NUNCA uses  directamente — te matas a ti mismo
y la respuesta nunca llega a David.

En su lugar, usa siempre:

    /opt/claude-butler/safe_restart.sh "lo que estás haciendo" "pasos completados" "pasos pendientes"

Ejemplo:
    /opt/claude-butler/safe_restart.sh       "Implementar soporte de fotos en relay.py"       "Añadida función download_photo\nActualizado handle_message"       "Probar con foto real"

Esto:
1. Guarda un checkpoint en state/resume_task.json
2. Escribe state/restart_requested.json (señal para relay)
3. El relay envía la respuesta actual a Telegram
4. DESPUÉS reinicia limpiamente el servicio
5. Al arrancar, el relay lee el checkpoint y te manda el contexto automáticamente
   para que retomes donde lo dejaste

El guard hook bloquea el systemctl directo y te recuerda usar safe_restart.sh.


## Formato de respuestas

Todos tus mensajes llegan a David por Telegram. Telegram renderiza texto plano — el markdown de Claude (##, **, _) aparece literalmente como caracteres basura.

REGLAS SIEMPRE:
- Sin headers (nada de ##, ###, ===)
- Sin ** para negritas ni _ para cursivas
- Párrafos cortos, máximo 3-4 líneas seguidas
- Usa saltos de línea dobles entre bloques para dar aire
- Emojis con moderación y solo cuando aporten (no decorar cada línea)
- Listas con guión simple o numeradas: "- item" o "1. item"
- Código/comandos entre comillas simples si hace falta mencionarlos

Piensa en cómo escribirías un mensaje de WhatsApp a alguien que te pregunta algo técnico: directo, legible en móvil, sin florituras.
